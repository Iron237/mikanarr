# 下载文件保持原样,番剧库为虚拟视图

> **⚠️ 已被 [ADR-0002](0002-organize-via-inplace-rename.md) 修订(2026-06-13)。**
> 不再坚持"绝不重命名":改为用 qB `renameFile` **原地重命名**整理成 Jellyfin 结构(保种、不复制、不硬链接),可开关。
> 本文档的"虚拟库""标题解析是核心""不移动/不复制"原则仍然成立;变化的只是"允许原地改名 + 写 NFO 落盘",详见 ADR-0002。

> **⚠️ 再修订:BDrip 覆盖 web 时删除 web(2026-06-17)。**
> 新增对"永不删除"的**刻意例外**:当一部番剧达成「全 BD 完成」(`eps_total` 已知 且正片 1..N 每集都有 `is_active` 的 BD 文件)时,系统会删除被 BD 顶替的 web 种子(qB 任务 + 磁盘文件)。详见下方「例外」一节;原盘 BD(`raw_disc`/`bd_owned`)不触发,web 永远保留作正片。

下载完成的视频文件保持字幕组原始命名留在 qBittorrent 下载目录中持续做种,系统不重命名、不移动、不硬链接;WebUI 的番剧库完全由数据库中「文件 ↔ 番剧/剧集」映射关系渲染,映射由种子标题解析建立。

原因:(1) 当前架构是 Win PC 上的 Docker 通过 SMB 写入 NAS,跨 SMB 创建硬链接不可行,AutoBangumi 式硬链接整理走不通;(2) 移动/重命名会中断做种,复制会翻倍占用空间;(3) 用户不依赖 Jellyfin/Plex,展示需求全部由本系统 WebUI 满足。

## Consequences

- 种子标题解析(标题 → 番剧 + 集数)成为核心组件,库视图、推送文案、去重都依赖它。
- 「整理模块」(硬链接出规整目录)作为可选功能预留,待系统迁移到 NAS 本机 Docker(同文件系统)后才可启用。

## 例外:BDrip 覆盖 web 时删除 web(2026-06-17,grill 第七轮)

番剧生命周期定为:web 追番 →(完结)智能补全 →(出 BD)BDrip 覆盖 web。用户明确选择「BDrip 覆盖 = 彻底替换」而非「仅隐藏」:补全完成后删掉冗余 web,腾盘。这与本 ADR「文件不动」冲突,作为**有意识的局部例外**记录于此。

**触发(`app/services/lifecycle.finalize_complete`):** 仅当「全 BD 完成」——`eps_total` 已知 **且** 正片 1..N 每集都有 `is_active` 的 BD 文件。`eps_total` 未知则永不触发(宁可不自动也不误判)。

**动作(一次性、幂等):** 清番剧 `auto_best`(停定期智能扫描)→ 停用该番真字幕组 RSS 订阅 → 删被 BD 顶替的 web 种子:`info_hash` 存在(真下载的)、其文件**全部非 active**(已被 BD 顶替)、且**不含 BD 源**(不误删 BD 自身)。合集种子必须整包都被顶替才删 → 不破坏种子完整性。BD 种子保留做种。

**不触发者:** 原盘 BD(`raw_disc` / `bangumi.bd_owned`)是收藏,不拆集、不进剧集网格,web 永远保留作正片,只跳 PowerDVD(见 ADR-0004)。本地导入/库容器(无 `info_hash`)的 web 文件也不在自动删除范围(只删"真种子")。

**权衡:** 拿"可回滚 + 保种率"换"腾盘 + 彻底替换"。删除不可逆、会掉种;故只在 BD 全集覆盖确认后整种子删除,把风险压到最低。
